In Fall 2024, I took Binary Exploitation Lab at Georgia Tech as part of my online Master’s. I had no CTF experience going in, no background in pentesting or red teaming. The course expects you to reverse engineer binaries, often without source code, find vulnerabilities in them, and exploit them. Coming in cold, it is tough.

Anyone learning exploitation will encounter the buffer overflow early. I was no different. To find exactly where in your input the return address sits, you reach for pwntools’ cyclic() and cyclic_find(). I used both throughout the course without once stopping to look at how they actually work. There was never time for that.

Here is the context.

When a C function is called, the CPU saves a return address on the stack: the memory location to jump back to when the function finishes. A typical stack frame looks like this:

Consider this vulnerable program:

#include <unistd.h>

void vuln() {
    char buf[40];
    read(0, buf, 200);   /* intentionally oversized */
}

int main() {
    vuln();
    return 0;
}

Compile it as a 32-bit binary with stack protections disabled:

gcc -m32 -o vuln vuln.c -fno-stack-protector -no-pie

read(0, buf, 200) asks the kernel to read up to 200 bytes from stdin into a 40-byte buffer. Feed it more than 40 bytes and it writes past the buffer, past the saved frame pointer, and over the return address. When vuln() tries to return, the CPU jumps to whatever bytes you supplied there.

Try sending 100 ‘A’s (0x41 in ASCII):

gdb confirms exactly what happened:

Program received signal SIGSEGV, Segmentation fault.
0x41414141 in ?? ()

The A’s walked right over the return address. You control where the program returns. The catch is you need to know exactly which byte of your input landed there. If you sent 100 identical ‘A’s, you cannot tell — they all look the same in the crash.

This is the offset problem. Anyone who has worked through a binary exploitation tutorial or a CTF challenge will have seen pwntools’ cyclic() used for exactly this:

from pwn import *

pattern = cyclic(100)
# aaaabaaacaaadaaaeaaafaaagaaahaaaiaaajaaakaaalaaama...

Feed that pattern to the binary instead of ‘A’s, crash it, read whatever bytes ended up in the instruction pointer, and:

offset = cyclic_find(b'laaa')
# 44

Offset 44 means the return address sits 44 bytes into your input: 40 bytes of buffer plus the 4-byte saved frame pointer that lives between the buffer and the return address on the stack.

With the offset known, you craft your input precisely. The first 44 bytes are padding, and the next 4 bytes are an address of your choosing. When vuln() returns, the CPU reads those 4 bytes as the return address and jumps there. If you point it at code you control, such as malicious instructions you placed earlier in the buffer, the CPU executes them instead of returning normally. That is the exploit.

But what is this pattern, exactly? It looks almost random (aaaabaaacaaad...). The usual explanation goes something like this: every group of four consecutive characters is unique, so whatever four bytes end up in the instruction pointer after a crash, you know exactly where in your input they came from.

That much I understood. What I never thought to question was how pwntools actually generates such a sequence, or why something like it can even exist in the first place: a single string that visits every possible four-character combination exactly once. I just used it without asking.

After the course, I finally had a chance to think about it. Here is what I found.

The sequence generated by cyclic() is called a de Bruijn sequence. Let’s get into what that means.

The Curious Bit String#

Before looking at the code, consider this 8-bit string:

00011101

Take all contiguous substrings of length three, wrapping around cyclically:

PositionSubstring
0000
1001
2011
3111
4110
5101
6010
7100

All eight substrings are distinct, and together they are exactly all eight possible 3-bit binary strings. This short string somehow manages to contain every possible 3-bit pattern exactly once, each starting at a different position.

This is exactly the property pwntools exploits. If every $n$-character window in a sequence is unique, reading back any $n$-character chunk immediately tells you its position. Pwntools defaults to $n = 4$; even on 64-bit targets, recovering four consecutive pattern bytes is usually enough to identify the offset.

A brief aside on $n > 4$: you will almost certainly never need it. With 26 lowercase letters and $n = 4$, the sequence is $26^4 = 456{,}976$ characters long, comfortably larger than any stack buffer you are likely to encounter. Unless the overflow sits 4 GB into the input, like CVE-2018-16301, which would not overflow until you fed tcpdump that much. For everything else, cyclic(200) and a coffee will do.

It is natural to wonder whether strings with this property exist in general, not just for binary, not just for length 3. It turns out they do, and they have a name.

De Bruijn Sequences#

A de Bruijn sequence of order $n$ over an alphabet of size $k$, written $B(k, n)$, is a cyclic string of length $k^n$ in which every possible string of length $n$ over that alphabet appears exactly once as a contiguous substring.

Our 00011101 is $B(2, 3)$: a cyclic binary string of length $2^3 = 8$ containing all eight 3-bit strings exactly once.

The pwntools cyclic() function produces a de Bruijn sequence over the 26-character lowercase alphabet with $n = 4$ by default, giving a cyclic sequence of length $26^4 = 456{,}976$ characters in which every 4-letter window is unique. Since exploit buffers are rarely that large, any prefix you need is sufficient.

These sequences are named after the Dutch mathematician Nicolaas Govert de Bruijn, who studied them in 1946, though the binary case was first proved by Camille Flye Sainte-Marie in 1894, who appears to have received considerably less credit for it.

Why they are optimal#

Here is a clean lower-bound argument. Any sequence containing all $k^n$ distinct length-$n$ strings must have those strings start at $k^n$ distinct positions (two substrings at the same position are identical by definition). So the sequence needs at least $k^n$ positions, meaning at least $k^n$ characters.

A de Bruijn sequence achieves this lower bound exactly — it is the shortest cyclic sequence in which every length-$n$ string appears, with each appearing exactly once. No wasted positions. No string skipped or repeated.

The Graph Theory Connection#

The existence of de Bruijn sequences for all $k$ and $n$ is not obvious. One elegant proof comes from graph theory.

The De Bruijn Graph#

Construct a directed graph as follows:

  • Vertices: all strings of length $n - 1$ over the alphabet. There are $k^{n-1}$ of them.
  • Edges: for each string of length $n$, draw a directed edge from its first $n - 1$ characters to its last $n - 1$ characters, labelled by the last character.

For the binary alphabet and $n = 3$, the vertices are $\{00, 01, 10, 11\}$ and the edges are:

000 → 00 --[0]--> 00
001 → 00 --[1]--> 01
010 → 01 --[0]--> 10
011 → 01 --[1]--> 11
100 → 10 --[0]--> 00
101 → 10 --[1]--> 01
110 → 11 --[0]--> 10
111 → 11 --[1]--> 11

Each vertex has exactly $k$ outgoing edges (one per possible next character) and exactly $k$ incoming edges. In-degree equals out-degree at every vertex.

Eulerian Circuits#

Königsberg, circa 1736. The city straddles the Pregel river; seven bridges (highlighted) connect four landmasses.

In 1736, the citizens of Königsberg posed themselves a puzzle: is there a walking route through the city that crosses each of the seven bridges exactly once and returns to the starting point?

Take a moment and try it mentally with the map above. After a couple of attempts, it should already start to feel like an impossible task.

Indeed, your intuition is correct. It is impossible, and this can be shown mathematically. Euler was the first to prove it. His key move was to abstract the geography away: represent each landmass as a node and each bridge as an edge. The walking-route question becomes a question about graphs.

Left: the physical layout with bridges numbered 1–7. Right: Euler’s abstraction, with landmasses A, B, C, D as nodes.

Euler showed that, in a connected graph, a closed walk traversing every edge exactly once (what we now call an Eulerian circuit) exists if and only if every node has an even number of edges. In Königsberg, every landmass connects to an odd number of bridges, so no solution exists. This seemingly recreational observation became the founding result of graph theory.

The relevance here: a directed graph has an Eulerian circuit if and only if:

  1. Every vertex has equal in-degree and out-degree.
  2. All vertices with nonzero degree lie in a single strongly connected component.

Both conditions hold for the de Bruijn graph. Every vertex has in-degree = out-degree = $k$, and the graph is strongly connected. Therefore, an Eulerian circuit always exists.

Now, why does an Eulerian circuit give us a de Bruijn sequence? Each edge corresponds to exactly one length-$n$ string: the edge 00 --[1]--> 01 corresponds to 001 (the source vertex gives the first $n - 1$ characters, the edge label gives the last). There are $k^n$ edges in total, one for every possible length-$n$ string.

As you walk the Eulerian circuit, consecutive edges overlap by $n - 1$ characters. When you traverse the edge spelling 001 and arrive at vertex 01, the next edge must leave 01, so whatever it spells starts with 01, sharing the last two characters of what you just read. Each new edge therefore contributes exactly one new character (its label), while the other $n - 1$ characters are already implied by which vertex you are standing on.

The result: you walk all $k^n$ edges, covering every length-$n$ string exactly once, and the string you read off has exactly $k^n$ characters. Every length-$n$ window in that string is distinct because each window corresponds to a different edge. That is precisely a de Bruijn sequence.

This also proves existence for all $k$ and $n$: the de Bruijn graph always has an Eulerian circuit, and any such circuit yields a de Bruijn sequence. The thing you needed to exist, which it was by no means obvious should exist, turns out to be guaranteed by a result Euler discovered while thinking about walking routes in a Prussian city.

The Algorithm#

Now let’s look at how pwntools actually generates the sequence:

import string

def de_bruijn(alphabet=string.ascii_lowercase, n=4):
    k = len(alphabet)
    a = [0] * k * n

    def db(t, p):
        if t > n:
            if n % p == 0:
                for j in range(1, p + 1):
                    yield alphabet[a[j]]
        else:
            a[t] = a[t - p]
            for c in db(t + 1, p):
                yield c

            for j in range(a[t - p] + 1, k):
                a[t] = j
                for c in db(t + 1, t):
                    yield c

    return db(1, 1)

This compact recursion generates the FKM construction, named after Fredricksen, Kessler, and Maiorana. Rather than finding an Eulerian circuit explicitly, it produces the lexicographically smallest de Bruijn sequence by concatenating Lyndon words in the required order. Martin’s 1934 paper gave an earlier construction by a different route.

What is a Lyndon word?#

A Lyndon word is a string that is strictly smaller, lexicographically, than all of its nontrivial rotations. In other words, it is the unique minimum element of its cyclic equivalence class.

Take the string aab. Its rotations are aab, aba, and baa. Since aab is smaller than both aba and baa, it is a Lyndon word. Now take aba. Its rotations are aba, baa, and aab. Since aab < aba, the string aba is not a Lyndon word; aab is the canonical representative of this cycle, not aba.

A few more examples over the alphabet {a, b}:

  • a — Lyndon ✓ (a single character is always Lyndon)
  • b — Lyndon ✓
  • ab — Lyndon ✓ (rotations: ab, ba; ab < ba)
  • ba — Not Lyndon (ab < ba)
  • aab — Lyndon ✓ (rotations: aab, aba, baa; aab is smallest)
  • abb — Lyndon ✓ (rotations: abb, bba, bab; abb is smallest)
  • aba — Not Lyndon (aab < aba)
  • abab — Not Lyndon (rotating by 2 gives abab again; a Lyndon word must be strictly smaller than every rotation, not equal to any)

From Lyndon words to de Bruijn sequences#

Here is the key theorem (Fredricksen and Maiorana, 1978): the lexicographically smallest $B(k, n)$ is obtained by concatenating, in lexicographic order, all Lyndon words whose length divides $n$.

Let us verify this for $B(2, 3)$ over the binary alphabet $\{0, 1\}$.

Find all Lyndon words with length dividing 3, that is, lengths 1 and 3:

  • Length 1: 0, 1
  • Length 3: 001, 011

(The other length-3 binary strings are not Lyndon: 010 has 001 as a smaller rotation; 100, 101, 110 all have smaller rotations; 000 and 111 equal their own rotations and so are not strictly smaller.)

Concatenate them in lexicographic order:

$$0 \;\; + \;\; 001 \;\; + \;\; 011 \;\; + \;\; 1 \;\; = \;\; 00010111$$

Slide a window of width 3 across 00010111 cyclically:

PositionWindow
0000
1001
2010
3101
4011
5111
6110
7100

All eight 3-bit strings appear exactly once. The Lyndon words slotted together perfectly.

The pwntools sequence aaaabaaac... is the same idea applied to the 26-letter alphabet with $n = 4$: all Lyndon words of lengths 1, 2, and 4 over the lowercase alphabet, concatenated in alphabetical order.

Why the algorithm avoids explicit enumeration#

The naive approach would be to generate all strings, test each for the Lyndon property, sort the ones that qualify, and concatenate. Workable for small cases, but expensive for large $k$ and $n$.

The FKM algorithm avoids this entirely. The recursive function db(t, p) builds candidate strings of length $n$ character by character into the array a. The parameter p tracks the period of the current candidate: the length of the Lyndon word being used as a repeating template.

The line a[t] = a[t - p] extends the current string by copying from its own earlier characters with period $p$, maintaining the template’s structure. When the recursion has filled $n$ positions (t > n) and the period $p$ divides $n$ exactly (n % p == 0), the first $p$ characters of a form a valid Lyndon word of the right length; output them. Then the inner loop for j in range(a[t - p] + 1, k) sets position t to a strictly larger character, breaking the current period and starting a new, lexicographically larger Lyndon word candidate. The recursion backtracks and tries all such extensions in order.

The effect: all Lyndon words of lengths dividing $n$ are enumerated in lexicographic order, one character at a time, without ever materialising a list of candidates. Each character is output exactly once.

The result is a generator that streams the de Bruijn sequence using $O(kn)$ auxiliary memory ($O(n)$ for pwntools’ fixed alphabet). Which is fortunate, because $26^4$ characters is already uncomfortable, and $26^8$ would be approximately 209 billion.

Complexity and Optimality#

Generation#

The de Bruijn sequence $B(k, n)$ has length $k^n$. Any correct algorithm must output $k^n$ characters, so $\Omega(k^n)$ is an unconditional lower bound on generation time: you cannot do better than this no matter how clever you are, because writing the output alone takes that many steps.

The FKM algorithm generates successive symbols with constant amortised work, so producing the complete $k^n$-character sequence takes $O(k^n)$ time. The only working storage is the array a of length $kn$ ($O(n)$ for pwntools’ fixed 26-letter alphabet). The generation algorithm is therefore time-optimal for the task of producing a complete cyclic cover. The FKM algorithm achieves this; the Lyndon word theory underlying it goes back to Fredricksen and Maiorana (1978).

Decoding#

Decoding is the inverse problem: given a substring, find its position. The pwntools source is admirably self-deprecating about this:

def cyclic_find(subseq, alphabet=string.ascii_lowercase, n=None):
    # ...
    # TODO: "Calculates" is an overstatement. It simply traverses the list.
    #
    # There exists better algorithms for this, but they depend on generating
    # the De Bruijn sequence in another fashion.
    #
    # https://www.sciencedirect.com/science/article/pii/S0012365X00001175

The current implementation walks the de Bruijn sequence from the beginning until it finds the target substring: $O(k^n)$ time. Which, for $n = 4$, is instantaneous in practice. But it acknowledges that it is leaving something on the table.

Jonathan Tuliani’s “De Bruijn sequences with efficient decoding algorithms” (Discrete Mathematics, 2001) addresses this head-on. Tuliani constructs de Bruijn sequences for which the position of any length-$n$ substring can be computed directly without traversing the full sequence. Tuliani takes a different route: instead of the lexicographically minimal FKM sequence used by pwntools, he constructs de Bruijn sequences specifically designed for efficient decoding.

For the typical case in exploitation ($k = 26$, $n = 4$, $k^n = 456{,}976$), the $O(k^n)$ traversal finishes before you notice it started. But the gap between pwntools’ approach and Tuliani’s is a useful reminder that “good enough for every practical case” and “theoretically optimal” are different things.

Putting It All Together#

You send a cyclic pattern, crash the binary, read back four bytes, call cyclic_find(), get the offset, write the exploit.

Under that workflow: cyclic() produces the de Bruijn sequence $B(26, 4)$, a $456{,}976$-character string where every 4-letter window is unique. It exists because the de Bruijn graph always has an Eulerian circuit (every vertex has equal in-degree and out-degree). It is generated by the FKM algorithm (Fredricksen, Kessler, Maiorana) in provably optimal $O(k^n)$ time using $O(kn)$ space. The decoding in cyclic_find() is a full-sequence traversal; Tuliani showed that different de Bruijn constructions can support efficient direct decoding, though this has not been implemented in pwntools.

That is what cyclic(100) is doing under the hood.
